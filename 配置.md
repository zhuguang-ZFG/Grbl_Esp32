
## 写字机硬件与固件配置说明
### 1. 主控与固件
- **主控芯片**: ESP32‑WROOM‑32（ESP32‑D0WDQ6，晶振 40MHz）  
- **固件**: Grbl_ESP32（3 轴写字机，基于 `custom_3axis_hr4988.h` 机器配置）  
- **开机信息**（串口上电日志中可见）：  
  - 固件版本与 SDK：  
    - `Grbl_ESP32 Ver ...`  
    - `Compiled with ESP32 SDK:...`  
  - **芯片唯一 ID**（用于授权/加密绑定）：  
    - 由 `machine_init()` 打印：  
      - `[ESP32ID] ChipID (efuse MAC) = %08X%08X`  
    - 建议上位机在初始化阶段读取并缓存此 ID，后续可用作程序加密或授权判断的硬件指纹。
---
### 2. 轴向步进电机配置
- **X 轴**  
  - STEP: `GPIO2`  
  - DIR : `GPIO15`  
- **Y 轴（Y1+Y2 两个电机并联，硬件共用 STEP/DIR）**  
  - STEP: `GPIO13`  
  - DIR : `GPIO12`  
- **Z 轴（抬笔）**  
  - STEP: `GPIO14` (MTMS)  
  - DIR : `GPIO27`  
> 说明：Y 轴是双电机并联接在同一套 STEP/DIR 上；Z 轴只负责抬笔。  
> **Z 轴开机默认在原点**：机械上为抬笔方向极限，系统认为 **Z=0**；落笔为 Z 正向（如 Z30）。
---
### 3. HR4988 驱动配置
- **参考电压（REF）**：纸路三电机（拾落/面板/进纸器）共用一根 REF，接 `GPIO25`（DAC）。固件会**按当前运行的电机**切换 DAC 值，实现每电机独立电流设定（见下方「每电机 REF」）。
- **使能脚（nENABLE，低电平=使能）**  
  - **主 XYZ 三轴**: `GPIO4`  
  - **拾落电机 + 进纸器电机**: `GPIO26`  
  - **面板电机**: 通过 74HC595 的 `Q1`（QB，对应逻辑引脚 `I2SO(1)`）
---
### 4. 74HC595D 扩展（换纸系统）
#### 4.1 控制信号连接（ESP32 → 74HC595D）
- **SI（数据输入，Pin 14）**: `GPIO21`  
- **SRCLK（移位时钟，Pin 11）**: `GPIO16`  
- **RCLK（存储时钟/锁存，Pin 12）**: `GPIO17`  
- **SRCLR（复位，Pin 10）**: 接 VCC（高电平，不复位）  
- **OE（输出使能，Pin 13）**: 接 GND（低电平=输出使能）
#### 4.2 74HC595D 输出位定义（Q0..Q7）
- **Q0 (`Pin 15`)**: 一键换纸按钮彩灯（固件已按状态驱动）  
  - HIGH = 灯灭，LOW = 灯亮  
  - **状态含义**：  
    - **常亮**：空闲且有纸（就绪，可按键换纸）或运行中有纸  
    - **慢闪（约 0.5s 周期）**：空闲且无纸（提示加纸）  
    - **快闪（约 80ms 周期）**：一键换纸流程执行中  
- **Q1 (`Pin 1`)**: 面板电机 HR4988 使能（nENABLE）  
  - LOW = 使能  
  - HIGH = 禁用  
- **Q2 (`Pin 2`)**: 拾落电机 DIR（方向，接 HR4988 Pin 19）  
- **Q3 (`Pin 3`)**: 拾落电机 STEP（步进，接 HR4988 Pin 16）  
- **Q4 (`Pin 4`)**: 面板电机 DIR（方向，接 HR4988 Pin 19）  
- **Q5 (`Pin 5`)**: 面板电机 STEP（步进，接 HR4988 Pin 16）  
- **Q6 (`Pin 6`)**: 进纸器电机 DIR（方向，接 HR4988 Pin 19）  
- **Q7 (`Pin 7`)**: 进纸器电机 STEP（步进，接 HR4988 Pin 16）
> 固件中对应逻辑引脚为：  
> - `I2SO(0)~I2SO(7)` 分别对应 Q0~Q7  
> - 拾落：DIR=`I2SO(2)`，STEP=`I2SO(3)`  
> - 面板：DIR=`I2SO(4)`，STEP=`I2SO(5)`  
> - 进纸：DIR=`I2SO(6)`，STEP=`I2SO(7)`
---
### 5. 传感器与按钮
- **纸张检测传感器**: `GPIO34`  
  - 逻辑：**LOW = 有纸，HIGH = 无纸**  
  - 当前配置中**不再映射为 Macro0 按钮**，故不出现在 `?` 指令的 `|Pn:` 中  
  - 状态通过专用命令查询：`M701`、`M704` 或 `[ESP901]`
- **一键换纸按钮**: `GPIO35`  
  - 低电平 = 按下（需要外接下拉或合理上拉电路）  
  - 映射为 `MACRO_BUTTON_0_PIN`（Macro0），并通过 `INVERT_CONTROL_PIN_MASK` 设为**低电平有效**  
  - **软件去抖**：在 `Custom/paper_system.cpp::user_defined_macro()` 中实现  
    - 仅在 `sys.state == Idle` 时响应按键  
    - 两次有效触发之间至少间隔 **500ms**（`millis()` 节流），防止抖动和误连按  
    - 每次有效触发会打印：  
      - 第一次按下：`[PaperBtn] Press again within 2.0s to change paper`；0.5～2s 内再按一次：`[PaperBtn] Triggered (double press), queuing [ESP910]...`  
    - 按键触发的换纸实际是向输入缓冲注入一条 `[ESP910]`，走与 Web 指令完全一致的自动换纸流程。
---
### 6. 换纸相关命令
#### 6.1 推荐：专用换纸命令（[ESP9xx]）
这些命令通过 Web/串口统一入口（非纯 GCode），推荐用于调试和点动控制：
- **查询纸张+电机状态**  
  ```text
  [ESP901]
返回类似：Paper=OK/No MotorEn=On/Off

- **拾落 / 面板 / 进纸器电机点动**（可带步数，默认 200 步，便于单独调试）  
  ```text
  [ESP911]        ; 拾落电机 200 步
  [ESP911 50]     ; 拾落电机 50 步
  [ESP912]        ; 面板电机 200 步
  [ESP912 100]    ; 面板电机 100 步
  [ESP913]        ; 进纸器电机 200 步
  [ESP913 80]     ; 进纸器电机 80 步
  ```
- **仅使能换纸驱动（不动作）**，便于先设方向再点动：  
  ```text
  [ESP930]
  ```
  执行后 MotorEn=On，可用 `M64 P1`/`M65 P1` 等设方向，再用 `[ESP911]`/`[ESP912]`/`[ESP913]` 点动。

串口或 Web 控制台里原样输入（包括方括号），回车即可，例如发 [ESP901] 会回复一行状态和 ok。

6.1.2 一键自动换纸（推荐）

- **Web/串口命令**：  
  ```text
  [ESP910]
  ```  
  或**连按两次**换纸键（GPIO35）：第一次按下串口提示“再按一次”，0.5～2s 内再按才执行 [ESP910]，防止误触。

- **自动换纸整体流程概述**（步数和方向可在 `custom_3axis_hr4988.h` 中调整宏）：  
  1. **使能换纸相关驱动（互斥两组）**  
     - 切换 I2S 为静态模式（passthrough）  
     - 面板组（Q1 低）：只在需要时单独使能  
     - 拾落+进纸组（GPIO26 低）：只在需要时单独使能  
  2. **面板电机弹出旧纸**  
     - DIR = `PANEL_DIR_EJECT`  
     - 运行 `PANEL_EJECT_STEPS` 步（约等于一张 A4 长度 + 余量，可调）  
  3. **进纸器送入新纸直到纸张传感器触发**  
     - DIR = `FEEDER_DIR_FORWARD`  
     - 单步前进，配合 `paper_sensor_stable()`（3 次采样、100µs 间隔）判断“有纸”，上限 `FEEDER_FIND_STEPS_MAX`  
  4. **松开拾落电机**  
     - DIR = `CLAMP_DIR_RELEASE`  
     - STEP = `CLAMP_TOGGLE_STEPS`（默认约 470 步，可调）  
  5. **拾落抬起后面板与进纸器同速送纸**（先送 6cm，然后面板+进纸器停止，仅拾落夹紧；夹紧完成后再同速送纸至 8cm，送纸器停止）  
     - 面板 DIR = `PANEL_DIR_FEED`，进纸器 DIR = `FEEDER_DIR_FORWARD`  
     - 先同步送纸 **`PAPER_ADVANCE_CM_CLAMP_START`×`PAPER_STEPS_PER_CM`** 步（默认 6cm）  
  6. **再次压紧拾落电机**（夹紧过程中面板电机、进纸器电机均停止运动，只步进拾落）  
     - DIR = `CLAMP_DIR_CLAMP`，STEP = `CLAMP_TOGGLE_STEPS`  
     - 夹紧完成后，面板+进纸器**再次同速**送纸至总长 **`PAPER_ADVANCE_CM`**（默认 8cm），然后送纸器停止、仅保留面板  
  7. **面板电机快速送纸，直到传感器“看不到纸”**（拾落夹紧后进纸速度加倍）
     - 仅面板电机使能，DIR = `PANEL_DIR_FEED`
     - 脉宽使用 `PAPER_PANEL_FAST_*`（头文件默认 15µs），前 `PAPER_PANEL_FAST_RAMP_STEPS` 步缓起步后加速，单步前进，上限 `PANEL_FAST_STEPS_MAX`
  8. **面板电机按 `PANEL_DIR_REVERSE` 回找传感器**
     - 仅面板电机使能，DIR = `PANEL_DIR_REVERSE`  
     - 单步回退，直到“再次感应到纸”或达到 `PANEL_BACK_STEPS_MAX` 上限  
  9. **面板电机再向送纸方向走固定步数，完成最终对位**（同步骤 7，夹紧后速度加倍）
     - DIR = `PANEL_DIR_FEED`
     - STEP = `PANEL_FINAL_STEPS`，脉宽使用 `PAPER_PANEL_FAST_*`（前 `PAPER_PANEL_FAST_RAMP_STEPS` 步缓起步）
  10. **关闭换纸相关电机使能**  
      - 拉高 Q1 和 GPIO26，防止三路纸路电机长时间发热  
      - 串口输出 `[PaperAuto] All steps completed successfully!` 及一行 **`[PaperStatus] N`**（见下）。  
  - **不运动时失能**：流程**开始**会先 `paper_disable_drivers()` 再按步骤使能当前需要的电机；步骤 2 进纸超时提前退出也会失能；流程结束或 M711 点动结束均会失能，换纸电机仅在运动时使能。
  - **XYZ 写完一页后失能**：M30 或回原点（G0 X0 Y0 / G28/G30）触发换纸时，固件在 `protocol_buffer_synchronize()` 后**立即**调用 `motors_set_disable(true)` 失能 XYZ，不依赖主循环 250ms 延时（换纸期间主循环被阻塞，否则会一直未失能）。

- **换纸结束状态码 `[PaperStatus] N`**（上位机可解析此行做分支）：  
  | N | 含义 |
  |---|------|
  | 0 | 成功完成全流程 |
  | 1 | 保留（未使用） |
  | 2 | 进纸阶段在限定步数内未触发传感器，已中止 |
  | 3 | 流程跑完，但第 7 步回找后传感器未稳定（纸可能未到位，仅警告） |

- **开始队列书写前出一次旧纸**：  
  - 设计上：上位机在**发送第一页 GCode 之前**先发 **`M721`**，固件会自动执行一次完整换纸（出旧纸 + 进第一张并对位），再回 `ok`。  
  - **若上位机不会发送 M721**（如奎享雕刻当前不支持）：需在开始队列前**手动**执行一次 **`[ESP910]`** 或按**一键换纸键**，效果相同；然后再由上位机发送第一页、第二页…（每页末尾 M30 自动换纸）。

- **可调参数汇总**（均在 `custom_3axis_hr4988.h`）：
  - **$ 参数默认值**：头文件 “Default Settings” 段仅覆盖本机相关项（如 $0=10、休闲 250、Z 方向反相、X/Y/Z 步数 200/200/400、速度 5000/5000/1000、加速度 500/500/400、行程 200/200/20）；其余 $ 使用 Grbl 的 Defaults.h。若需与串口 $ 完全一致，可在头文件自行添加对应 `DEFAULT_*` 宏。
  - 步数：`PANEL_EJECT_STEPS`、`PANEL_FAST_STEPS_MAX`、`PANEL_BACK_STEPS_MAX`、`PANEL_FINAL_STEPS`、`FEEDER_FIND_STEPS_MAX`、`FEEDER_EXTRA_STEPS`；进纸 **`PAPER_ADVANCE_CM_CLAMP_START`**（默认 6cm）开始夹紧，总送纸 **`PAPER_ADVANCE_CM`**（默认 8cm）后送纸器停止；`CLAMP_TOGGLE_STEPS`
  - 步进脉宽（μs）：`PAPER_RAMP_STEPS`、`PAPER_RAMP_HI_US`/`PAPER_RAMP_LO_US`、`PAPER_NORMAL_HI_US`/`PAPER_NORMAL_LO_US`、`PAPER_CLAMP_HI_US`/`PAPER_CLAMP_LO_US`  
  - 方向：`PANEL_DIR_FEED`、`PANEL_DIR_EJECT`、`PANEL_DIR_REVERSE`、`FEEDER_DIR_FORWARD`、`CLAMP_DIR_*`  
  - **Z 轴点动限速**：`JOG_Z_MAX_FEED_MM_PER_MIN`（默认 1000 mm/min）。点动含 Z 向移动时（抬落笔）进给会被限制到此值，避免 $J= 用较大 F（如 2500）时 Z 过快；仅 XY 点动不受限。

6.3 换纸电机步进时序（卡顿/带载不足时可调）
- **脉宽由头文件宏控制**（`custom_3axis_hr4988.h`）：  
  - 面板/进纸器：起步 `PAPER_RAMP_STEPS` 步内为 `PAPER_RAMP_HI_US`/`PAPER_RAMP_LO_US`（默认 400µs），之后为 `PAPER_NORMAL_HI_US`/`PAPER_NORMAL_LO_US`（默认 150µs）。  
  - 拾落电机：`PAPER_CLAMP_HI_US`/`PAPER_CLAMP_LO_US`（默认 2000µs）。  
  - **拾落夹紧后面板进纸**（步骤 6、8）：`PAPER_PANEL_FAST_HI_US`/`PAPER_PANEL_FAST_LO_US`（头文件默认 15µs，约为正常 150µs 的约 10 倍速）；前 `PAPER_PANEL_FAST_RAMP_STEPS` 步仍用 400µs 缓起步（默认 10 步），之后才用快速脉宽，加速更早可见。  
- 若进纸器**仍卡顿**：优先检查 REF 电流、供电与接线；必要时将 `PAPER_NORMAL_*_US` 或 `PAPER_RAMP_*_US` 略增大。  
- **换纸中 Core 1 panic (Interrupt wdt timeout)**：多为步骤 2/6/7 等长循环未及时让出 CPU。固件已每 **50 步** `delay(1)` 一次（`PAPER_YIELD_STEPS=50`）；若仍复现可再减小该值或检查 I2S 是否处于 passthrough（stream 模式每步 delay 很长易超时）。
- **夹紧后面板加速不明显**：步骤 5 结束会切到仅面板并使 I2S 为 passthrough，此时每步延时约 8µs，快速脉宽才生效；若 I2S 处于 stream 模式会每步 delay(12ms)，加速感会消失。可把 `PAPER_PANEL_FAST_RAMP_STEPS` 改为 `5` 或 `0` 以更快进入高速；若 15µs 丢步可适当增大 `PAPER_PANEL_FAST_*_US`（如 50µs）。  
- **REF 直连 GPIO25（DAC）**：默认 `PAPER_DRIVER_REF_DAC=100`（≈1.3V）。**每电机可单独设定**：在 `custom_3axis_hr4988.h` 中定义 `PAPER_REF_DAC_CLAMP`、`PAPER_REF_DAC_PANEL`、`PAPER_REF_DAC_FEEDER`（0–255），不定义则用 `PAPER_DRIVER_REF_DAC`。直连时建议 80–150，发烫则调小对应电机值。  
- **面板电机启动时电路板断电/重启**：多为**电源被电机上电瞬间电流拉低**（掉压导致 ESP32 欠压复位）。处理建议：  
  1. **硬件**：控制板与电机/驱动**分开供电**（共地即可）；或换用电流更大的电源；在电机驱动附近加 **100–470µF 电解**缓冲。  
  2. **固件**：在 `custom_3axis_hr4988.h` 中定义 `PAPER_REF_SOFTSTART_MS`（如 `10`），使能电机时先 REF=0 再拉低 EN，延时该毫秒数后再把 REF 升到目标值，可明显减小上电冲击（设为 `0` 或不定义则关闭软启动）。

6.2.5 M30 / M02 自动换纸（分页写字，“每页一个文件”模式）

- **设计目标**：一页 GCode 执行到**程序结束码**时，固件先同步完本页所有运动，再**自动换纸**，换纸结束后才回复 `ok`。  
- **触发条件**：固件对 **`M30`** 和 **`M02`** 都会执行换纸（二者在 Grbl 里均会调用 `user_m30()`）。  
  - 若上位机（如奎享雕刻）在**每页/每文件结束时没有下发 M30 或 M02**，固件不会执行换纸，这是“队列结束没换纸”的常见原因。  
- **固件实现要点**：  
  - `GCode.cpp` 中，M30→`CompletedM30`、M02→`CompletedM2`，在执行 program flow 时都会：  
    1. `protocol_buffer_synchronize()`，确保本页运动执行完；  
    2. 重置部分 modal 状态；  
    3. 调用 `user_m30()`（`Custom/paper_system.cpp`）；  
  - `user_m30()` 仅在 Idle 且纸张系统已配置时调用 `paper_auto_change()`，换纸结束后才返回，随后该行才回 `ok`。  
- **奎享雕刻等上位机**：  
  - 若每页末尾是 **`G90 G0 X0 Y0`**（回原点），固件会在此行执行完后自动换纸，**无需**再加 M30（见 6.2.6）。  
  - 若每页末尾**不是**回原点，则**必须**在每页文件末尾包含 **`M30`** 或 **`M02`**，固件才会换纸。若软件不会自动加 M30，可：  
    1. 在生成/导出每页 GCode 时**手动在文件末尾加一行 `M30`**；或  
    2. 在奎享中查找是否有“每文件结束发送 M30”或“程序结束符”类设置并开启。  
  - 可用串口监控查看：每页发送完毕后，是否有一条以 `M30` 或 `M02` 结尾的指令；若没有，说明上位机未发程序结束码，需按上面方式补上。  
- **正确流程**：每页文件末尾有 M30（或 M02）→ 固件执行完本页并换纸 → 回 `ok` → 上位机再发下一页。

6.2.6 回原点后自动换纸（适配奎享雕刻等不发 M30 的上位机）

- **触发方式**：固件在检测到**回原点**指令并执行完该运动后，会自动执行一次换纸（与 M30 相同逻辑）。  
- **会触发换纸的指令**：  
  1. **G28**、**G30**（回预设原点）；  
  2. **G0 X0 Y0**（或 `G90 G0 X0 Y0`）：快速移动到当前工件坐标系 XY 零点。  
- **奎享雕刻**：若每页末尾是 **`G90 G0 X0 Y0`** 而不是 M30，固件会在此行运动完成后自动换纸，**无需**在每页文件里加 M30。  
- **实现**：在 `GCode.cpp` 中，**仅当本行实际执行了**回原点（G28/G30 或本行有 G0 且目标为工作坐标 (0,0)）时，才先 `protocol_buffer_synchronize()` 再 `user_m30()`。必须用“本行是否执行了 G0”标志，否则下一页首行（如 G21）会因 modal 仍为 Seek、position 仍为 (0,0) 而误触发换纸（表现为“队列开始后一直在换纸”）。工作坐标 (0,0) 容差 0.01mm。

- **关于奎享末尾行 `G90G0 X0Y0`**：  
  - Grbl 按“字母+数值”分词，`G90G0` 会解析为 **G90** 和 **G0**，`X0Y0` 会解析为 **X0** 和 **Y0**，因此理论上会触发“回原点换纸”。  
  - 若实际未换纸，可能原因：  
    1. **user_m30() 内部被跳过**：仅在 `sys.state == Idle` 且纸张系统已配置（`PAPER_SENSOR_PIN != 255`）时才执行换纸；否则串口会看到 `[PaperM30] Skipped auto-change: ...`。  
    2. **上位机未等本行 ok**：若奎享在发完 `G90G0 X0Y0` 后立刻发下一页或断开，固件仍会执行换纸，但可能看起来像“没反应”（可观察串口是否先出 `ok` 再出下一页）。  
  - **调试**：在机器头文件或 `Config.h` 中定义 **`#define DEBUG_PAPER_ORIGIN`** 后重新编译，串口会在检测到回原点时打印 `[PaperOrigin] Return-to-origin detected, syncing then paper change`，用于确认该行是否被识别。

- **每页 GCode 格式约定**（奎享 / 导出多页时建议统一）：  
  - **页首**：`G21`、`G90`，以及 `F1500`、`G1 G90 Z0.0`、`G4 P0.2` 等本机需要的初始化。  
  - **页尾**：**`G90 G0 X0 Y0`**（或 `G90G0 X0Y0`）。固件执行到该行会先回原点，再自动换纸，然后回 `ok`。  
  - 多页时：每页都是「页首 + 本页书写 + 页尾」，下一页紧接下一段「页首 + 书写 + 页尾」；页尾的 `G90 G0 X0 Y0` 即分页符，遇到一次换纸一次。

6.2.7 开始写字前自动换纸（无需 M721 / 手动键）

- **行为**：上电（或 Grbl 复位）后，固件**仅当第一次**执行 **G0 X0 Y0 Z0**（回原点，含 X/Y/Z 且目标为 0）时，先同步缓冲区、执行一次完整换纸（出旧纸 + 进第一张并对位）、延时 250ms，再执行该行；串口会打印 `[Paper] 1st page`。纯抬笔/落笔（如 G1 Z0、G1 Z30）或仅 XY 的 G0/G1 不触发。
- **效果**：队列开头若有 **G0 X0 Y0 Z0**，点“开始”后会自动换纸再回原点；第一页末尾用 `G90 G0 X0 Y0` 触发换纸，之后每页同理。若队列开头没有 G0 X0 Y0 Z0，需手动 [ESP910] 或 M721 做第一页前换纸。
- **同一上电周期内跑多个队列**：仅第一次遇到 G0 X0 Y0 Z0 时换纸；之后若再跑第二队列且开头无 G0 X0 Y0 Z0，需手动 [ESP910] 或 M721。
- **说明**：上述第一页/页末换纸及 Z 轴（开机 Z=0、换纸后 Z=0）逻辑结构已用他人程序验证，无问题。

6.2.8 M721「队列开始」：第一页前换纸（可选）

- **含义**：上位机在**发送第一页 GCode 之前**发 **`M721`**，固件执行一次完整换纸；与 6.2.7 的“第一次书写前自动换纸”二选一即可（固件已支持自动，M721 用于需精确控制时）。
- **用法**：在队列中第一个文件之前插入一行 `M721`，等收到 `ok` 后再发第一页内容。

#### 6.2.9 授权 M800（带密钥，固件与厂商算法一致）

- **作用**：仅当串口收到正确的 **M800 P\<授权码\>** 后，才允许 G0/G1 等运动；授权结果存 NVS，重启仍有效。
- **用户**：上电后串口打印 `[ESP32ID] ChipID = XXXXXXXX YYYYYYYY`，将这两个 8 位十六进制数发给厂商，收到厂商下发的 **M800 P\<十进制数\>** 后，在串口发送该行即可授权。
- **厂商**：用与固件**相同的密钥和算法**根据 ChipID 算出授权码（十进制），把 `M800 P<该数>` 发给用户。算法见下（伪代码，与 `Custom/paper_system.cpp` 中一致）。

**授权码算法（伪代码）**

- **输入**  
  - `ChipID_hi`：芯片 ID 高 32 位（`[ESP32ID] ChipID = XXXXXXXX YYYYYYYY` 的前 8 位十六进制转成 32 位无符号）。  
  - `ChipID_lo`：芯片 ID 低 32 位（后 8 位十六进制转成 32 位无符号）。  
  - `KEY_H`：固件中的 `LICENSE_SECRET_KEY_H`（当前为 `0x8B3C9A1Fu`）。  
  - `KEY_L`：固件中的 `LICENSE_SECRET_KEY_L`（当前为 `0xE72F4D06u`）。  
- **约定**：所有运算为 32 位无符号整数，溢出自然截断。

```
FUNCTION license_code( ChipID_hi, ChipID_lo, KEY_H, KEY_L ) -> U32

  mix := ChipID_hi XOR ChipID_lo XOR KEY_H XOR KEY_L

  mix := ROTATE_LEFT( mix, 7 )   // 即 (mix << 7) OR (mix >> 25)，仅取低 32 位

  mix := mix XOR KEY_H

  rot_L := ROTATE_LEFT( KEY_L, 13 )   // (KEY_L << 13) OR (KEY_L >> 19)
  mix   := mix XOR rot_L

  RETURN mix   // 十进制即 M800 P 的数值

END
```

- 厂商端用上述函数算出返回值，将**十进制**结果作为授权码，用户发送 **M800 P\<该十进制\>**。
- 若授权码为 9 位数（如 121261484），因 G 码 P 为浮点会丢精度，请用 **M800 P\<高段\> Q\<低段\>**（授权码 = P×1000 + Q）。例如 121261484 发送：**M800 P121261 Q484**。

## 8. 故障排除与修复记录

### 8.1 问题1：GPIO34 纸张传感器无法正常工作

**现象**：固件启动后执行 M701 或 M704 命令，显示 "Paper=OK MotorEn=On"，无论是否有纸都输出相同结果；传感器状态不变。

**原因**：GPIO34 从未使用 `pinMode(GPIO_NUM_34, INPUT)` 初始化，导致无法进行外部电压的正确采样，GPIO 引脚浮空或保持重置时的默认状态，固件逻辑无法感知实际电压变化。

**解决方案**（已实施 / 2026-03-12）：
- 在 `paper_system_init()` 函数中添加：
  ```cpp
  pinMode((int)PAPER_SENSOR_PIN, INPUT);
  int initial_value = digitalRead(PAPER_SENSOR_PIN);
  ```
- 同时读取并输出初始值用于启动时调试，确保传感器连接正确

**验证方法**：
1. 重新编译 & 上传固件
2. 执行 M701 或 M704 命令，应输出 GPIO34 原始值（0 或 1）
3. 插入/移除纸张，重新执行 M704，GPIO 值应改变
4. 若值仍不变，检查 GPIO34 硬件连接和传感器本身


---

### 8.2 问题2：自动升级 (Auto Download) 失败，无法进入 Bootloader

**现象**：上传固件时 PlatformIO 报错或卡住，需要手动按 IO0 + RESET 才能进入 bootloader；自动进入 bootloader 的机制失效。

**原因**：原始代码在 `paper_system_init()` 中过早调用 `i2s_out_set_passthrough()`，在系统启动的早期就初始化了 I2S 外设。**GPIO16 和 GPIO17 是 I2S 的 SRCLK 和 RCLK 脚线，在某些 ESP32 芯片版本上与内部 Flash SPI 或 USB 接口共享**；过早初始化 I2S 可能干扰这些脚线的状态，导致自动 bootloader 序列失败。

**解决方案**（已实施 / 2026-03-12）：
- 延迟 I2S passthrough 模式的设置，从启动时移到**第一次真正需要电机操作时**
- 具体改动：
  - `paper_system_init()` 现在**仅初始化 GPIO34（传感器）和 GPIO25（DAC）**，完全不涉及 I2S 操作
  - `paper_enable_drivers()` 中添加一次性标志，仅第一次调用时执行 `i2s_out_set_passthrough()`，之后跳过
  - I2S 硬件初始化仍由 `Grbl.cpp` 的全局初始化流程负责（不重复）

**原理**：  
系统启动时 `Grbl.cpp` 已完整初始化 I2S 硬件，但不必立刻切换 passthrough 模式。将 passthrough 模式设置推迟到执行电机命令时，可避免在 bootloader 决策期间对 GPIO16/17 进行额外处理。

**验证方法**：
1. 重新编译并尝试**自动上传**（不需要手动按按键）
2. 自动升级应恢复正常
3. 如仍有问题，用 esptool.py 手动检查是否真的进入 bootloader

**关键代码位置**：
- [PaperSystem.cpp](Grbl_Esp32/src/PaperSystem.cpp) 第 43–55 行（paper_system_init）
- [PaperSystem.cpp](Grbl_Esp32/src/PaperSystem.cpp) 第 84–101 行（paper_enable_drivers）

### 8.3.1 XY 轴运行中卡顿（书写时一顿一顿）

**常见原因与对应调整**（按优先级试）：

1. **拐角减速感（像每个小段顿一下）**  
   - **$11 Junction deviation** 过小：拐角处规划器会大幅降速再转弯，观感像卡顿。  
   - **试**：串口 `$11=0.02` 或 `0.03`（默认 0.01），`$S` 保存。若 GCode 由很多短直线组成，可试到 **0.05**，拐角更圆、减速更少。

2. **加速度偏小（起步/每段开头慢一下）**  
   - **$120（X）、$121（Y）** 过小：每段都要从较低速度加速，短段多时就像一直“刚起步就减速”。  
   - **试**：适当加大 X/Y 加速度（例如 3000→4000～5000 mm/s²），加速段变短、匀速段更长。注意过大会增加丢步风险，需结合电机/机械试。

3. **步进脉冲/方向时序**  
   - **$0 步进脉冲** 为 3µs（最小值）：部分驱动需 5～10µs 才稳定，脉宽不足可能偶发丢步，表现为间歇卡顿或越写越偏。  
   - **试**：`$0=5` 或 `$0=8`，`$S` 保存。  
   - **Stepper/Direction/Delay** 过小：方向变化后立刻发步可能丢步。头文件默认 15µs；若 XY 换向时特别容易顿，可串口调大该延时（如 20～25µs）后 `$S`。

4. **GCode 段太密（很多极短 G1）**  
   - 笔画被拆成大量短直线时，规划器会频繁加减速，看起来像连续小卡顿。  
   - **试**：生成 GCode 时适当加大“平滑/拟合”或“弧线精度”，减少短直线数量；或略增大 **$12 圆弧精度**（如 0.005）让圆弧替代部分短直线。

5. **供电/机械**  
   - XY 与 Z/换纸共电源时，电机同时动作导致电压跌落，可能偶发丢步或顿一下。可试单独供电、加大滤波电容。  
   - 机械阻力不均（导轨、同步带过紧、某段卡）也会表现为固定区域卡顿，需排查机械。

**建议排查顺序**：先试 **$11=0.03** 与 **$0=5**（无需重烧），再根据现象调 **$120/$121** 或方向延时；若仍卡顿再查 GCode 段密度与供电/机械。  

**根据串口 log 排查 XY 卡顿**（针对“大量短 G1 + Z 抬落笔”类 GCode）  
- **log 特征**：若 log 里大量连续短段，例如 `G1 X102.475Y-145.688` → `G1 X102.489Y-145.732`（XY 仅 0.01～0.05mm 级）、中间穿插 `G1G90 Z0.0` / `Z5.0` 抬落笔，且 **ok 经常逐行返回**（一行一个 ok），则：  
  1. **段过密**：规划器会在每个拐角降速再加速，观感就是“每段小停一下”或“拐角顿”。  
     - **试**：串口 **$11=0.03** 或 **0.05**（拐角少减速）、**$120/$121** 适当加大（如 4000～5000），`$S` 保存。  
  2. **缓冲易空**：ok 逐行返回说明多数时刻 planner 里只有 1 个 block，段缓冲容易吃紧，段边界易出现短暂“等段”的顿感。  
     - **试**：上位机尽量**预发多行**再等 ok（流式发送、一次多行），避免严格“发一行等一 ok”；若用蓝牙，可试串口线对比，排除蓝牙延迟导致的主循环被占、段补充不及时。  
  3. **Z 与 XY 混段**：Z 在 0/5 间频繁切换，每段都是 XYZ 三维拐角，$11 对拐角减速影响大。  
     - **试**：同上加大 **$11**；若头文件里 **$122（Z 最大速率）** 或 Z 加速度偏大，可适当减小，减轻 Z 段对整体节奏的冲击。  
- **运行中按换纸键导致卡顿**：若 log 里出现 **`[PaperBtn] Ignored: system not idle (state=4)`**，原先会发这条 MSG，串口/蓝牙发送会短暂阻塞主循环导致卡顿。固件已改为运行中按下时**仅静默忽略、不再发送**。
- **未按键却出现 Ignored（误触发）**：换纸键接 GPIO35，步进/EMI 可能使引脚产生毛刺，框架在 CHANGE 中断里单次读引脚即调回调，易误触发。固件已加强去抖：**稳定低电平确认**（约 8ms 内连续 5 次采样均为 LOW 才视为真实按下，任一次为 HIGH 则静默丢弃），并在 500ms 间隔去抖处静默忽略、不再打印 debounce MSG。

**固件默认**：头文件里 $0=3µs、$11=0.01、方向延时 15µs；若需验证缓解卡顿，可串口临时改 **$0=5**、**$11=0.03**、Stepper/Direction/Delay=20 后 **$S** 保存。

**关于“喂狗”**：**不是喂狗导致的**。XY 步进在定时器 ISR + RMT 硬件中执行，**步进路径里没有任何 delay/喂狗**。任务看门狗（Task WDT）的 `esp_task_wdt_reset()` 只在主循环的 `COMMANDS::handle()` → `wait(0)` 里调用，与步进 ISR 不同任务、不阻塞步进时序。换纸里的 `delay(1)`（每 50 步 yield）仅用于**纸路步进**，避免 Interrupt WDT 复位，**不参与 XY 轴**。若卡顿与网络/蓝牙同时明显，可优先怀疑同一主循环里的 Web/WiFi/蓝牙占用 CPU，而非喂狗本身。

**XY 轴停止时卡顿（到终点/每段结束那一下）**

若观察是**停下来的瞬间**顿一下，而不是运行中或拐角：

1. **“快到终点时慢慢拖一下再停”**  
   - 减速段尾巴太长，最后几毫秒速度很低，像拖一下再停。  
   - **试**：适当**加大** **$120（X）、$121（Y）** 加速度（如 3000→3500～4000），减速更短、停止更干脆。

2. **“到终点时猛顿一下”**  
   - 减速太急，停的瞬间惯性明显。  
   - **试**：适当**减小** **$120/$121**（如 3000→2500～2800），减速更平缓，停止更柔和。

3. **每段 G1 结束都像小停一下**  
   - GCode 被拆成很多短段时，每段结束都会减速到拐角再加速，观感像“每段停一下”。  
   - **试**：继续加大 **$11**（如 0.03→0.05），或优化 GCode 减少极短直线。

4. **停止后马上反向的那一下**  
   - 方向建立时间不足，停→反相→第一步易丢步或顿。  
   - **试**：把 **Stepper/Direction/Delay** 从 20µs 调到 **25µs**（串口改后 `$S`），或头文件 `STEP_PULSE_DELAY` 改为 25 后重烧。

**固件侧（已修）**：  
- 段准备时若因浮点舍入得到 **n_step=0**（非 hold），原先会入队导致 ISR 里 `step_count` 下溢、步进异常或观感卡顿。已在 **Stepper.cpp** 中对该情况强制 `n_step=1`。  
- 计算步进周期时，若分母 `(last_n_steps_remaining - step_dist_remaining)` 过小会导致 **inv_rate 爆炸**、单步周期被钳到 0xffff ticks，表现为减速/停止前一拍“顿一下”。已对分母做**下限 0.5** 钳位，避免单步周期过长。  
- **强制 n_step=1 的段**：若仍参与 AMASS 左移（n_step 变为 4/8），而该段每轴步数右移后可能为 0，会产生多拍“空步”、明显停顿。已对该类段**强制 amass_level=0**，不再参与 AMASS，避免空拍卡顿。

**卡顿可能原因汇总（固件侧）**  
| 原因 | 表现 | 处理 |
|------|------|------|
| n_step=0 入队 → step_count 下溢 | 减速/停止段异常步进或顿感 | 已修：强制 n_step=1 |
| inv_rate 分母过小 → isrPeriod=0xffff | 停前单步周期过长、“顿一下” | 已修：分母下限 0.5 |
| 强制 1 步的段被 AMASS 左移 + 每轴步数 0 | 多拍空步、一段明显停顿 | 已修：该段 amass_level=0 |
| 段缓冲被吃空（st_prep_buffer 调用不及时） | 运动暂停或断流式卡顿 | 主循环少做重活；关 WiFi/蓝牙试 |
| 段边界 ISR 内加载新段 | 每段首步有轻微时序抖动 | 当前设计如此，影响通常很小 |

**与 GitHub 上游 (bdring/Grbl_Esp32) 对比**  
- 上游仓库：<https://github.com/bdring/Grbl_Esp32>（本仓库已配置 `upstream` 指向该地址）。  
- **Stepper.cpp 段准备逻辑**：上游在 `n_step==0` 且非 hold 时**不做任何处理**，该段仍会入队（n_step=0），ISR 中 `step_count` 下溢；且上游**未**对 `(last_n_steps_remaining - step_dist_remaining)` 做下限钳位，也未对“强制 1 步”的段禁用 AMASS。因此**卡顿相关的 4 处修改均为本仓库独有**：  
  1. `n_step==0` 时强制 `n_step=1` 并设 `forced_one_step`  
  2. 计算 inv_rate 前对分母做 **≥0.5** 钳位  
  3. `forced_one_step` 时强制 `amass_level=0`  
- **其它差异**：换纸系统（PaperSystem.cpp、custom_3axis_hr4988.h、paper_system 等）、配置与故障排除文档（本文档）等均为本机定制，上游无对应文件或逻辑。  
- 若需与上游同步：可 `git fetch upstream` 后对 Stepper.cpp 做选择性合并，保留上述 4 处修改。

**X 轴卡顿排查结论**：固件中**未发现仅针对 X 轴**的逻辑错误；X/Y/Z 共用同一套 Bresenham、段缓冲与 RMT 步进流程，RMT 波形（方向延时、脉宽）由最后一轴 init 写入、三轴一致。若**只有 X 轴**卡顿，可优先查：X 轴 **GPIO2（STEP）/GPIO15（DIR）** 接线与干扰、该轴机械阻力或同步带、该路供电/驱动；若 XY 同时卡顿，仍以 **$11/$0/$120/$121**、方向延时及上述固件修正为主。段缓冲依赖协议循环频繁调用 `st_prep_buffer()`，主循环被 Web/WiFi/蓝牙等占满时也可能造成短暂断流式卡顿。

先区分是“拖一下再停”还是“猛停一下”，再按上面 1 或 2 调 **$120/$121** 验证。

### 8.3 问题3：写完最后一张没排出去 / 刚开始写时 Z 轴卡顿

**现象 1：队列最后一页写完后，最后那张纸没有排出去**  
- 先看串口：最后一页的 `G90 G0 X0 Y0` 之后是否出现 **`[PaperAuto-1] Ejecting old paper (12000 steps)...`**。  
  - **有**：说明换纸已触发，Step1 在执行。若纸仍没出来，多半是**步数不够**或**方向反了**。在 `custom_3axis_hr4988.h` 里：  
    - 加大 **`PANEL_EJECT_STEPS`**（默认 12000），例如改为 15000～18000 试到能完整排出；  
    - 若纸往反方向走，把 **`PANEL_DIR_EJECT`** 取反（`true`↔`false`）。  
  - **没有**：说明最后一页没有触发换纸，检查该页末尾是否真有 `G90 G0 X0 Y0`。

**现象 2：刚开始写时 Z 轴卡顿一下**  
- 固件已在“第一页换纸”结束后加 **250ms 延时** 再执行第一条书写；若确认卡顿与换纸无关，见下条。

**Z 轴卡顿 / 方向反了（可逐项排除）**  
- **“先反一下再正”**：固件已把 **$Stepper/Direction/Delay** 默认设为 **15µs**（`STEP_PULSE_DELAY`）；若仍卡顿，可试 **20～25µs**（串口发 `$` 查当前值，改 Stepper/Direction/Delay 后 `$S` 保存，无需重烧）。  
- **Z 方向与 GCode 相反**：固件默认已对 Z 轴方向反相（`DEFAULT_DIRECTION_INVERT_MASK = bit(Z_AXIS)`）。若实际抬笔/落笔仍反，串口发 **$3=0** 再 **$S** 取消 Z 反相；反之若需反相则 **$3=4** 再 **$S**。  
- **接线**：Z 轴 STEP=GPIO14(MTMS)、DIR=GPIO27；确认无虚接、与 74HC595/I2S 无串扰。  
- **供电**：Z 与 XY/换纸电机共电源时，换纸瞬间电压跌落可能导致 Z 顿一下；可试单独供电或加大滤波。  
- **参数**：`$120`（Z 加速度）、`$122`（Z 最大速率）过大可导致起步过猛；可适当减小试机。  
- **点动限速**：Z 点动已用 `JOG_Z_MAX_FEED_MM_PER_MIN` 限速；若卡顿只在 GCode 时出现，可对比同速度点动是否也卡。  

**固件侧已核对（尚未发现可解释“卡顿”的明显错误）**  
- **Stepper ISR**：新 segment 加载时先更新 `dir_outbits`，下一拍再发步进；块边界处未见“多发一步/反方向一步”的逻辑错误。  
- **可能现象**：若卡顿是“起步那一下特别慢，然后正常”，多半是**加速段**：从静止到 F3000 的前几毫秒，Planner 会生成步数很少、周期较长的 segment，看起来像顿一下再走。可试把 **$120（Z 加速度）** 适当加大（例如 1200～1500），让加速段更短，看观感是否改善。  
- **进一步排查**：用示波器看 Z 的 STEP/DIR 是否有一次反向脉冲、或首步脉宽异常；或临时在 Z 步进时拉高某 GPIO 用 LED 观察节奏，区分“多/错一步”与“起步慢”。

- 若后续找到原因，可把结论补写在本节或 8.4 调试记录中。

### 8.4 调试与未来改进

若后续发现类似的初期化顺序问题，优先原则：
1. **硬件初始化** → 传感器、DAC、简单 GPIO（早期）
2. **复杂外设初始化**（I2S、DMA、中断）→  由全局框架统一管理
3. **功能模式开启**（passthrough、PWM、特定任务）→ 延迟到实际需要时

此原则可有效避免 bootloader 干扰和启动时序问题。

### 8.5 问题5：开机电流过大 / 上电断电

**现象**：上电或重启瞬间电流很大，电源保护或断电。

**原因**：主 XYZ 的全局使能脚（GPIO4，nENABLE 低有效）在 `motors_init()` 里只做了 `pinMode(OUTPUT)`，未写入“禁用”电平；ESP32 默认输出 0（LOW）= 使能，导致**上电瞬间 XYZ 三轴同时使能**，换纸电机由 paper_system_init 置为禁用，但主三轴在约 250ms 内一直处于使能，电流叠加易触发保护。

**解决方案**（已实施）：
- 在 `Motors.cpp` 的 `motors_init()` 中，对全局使能脚 `pinMode` 后立即执行 `digitalWrite(STEPPERS_DISABLE_PIN, step_enable_invert->get() ? 0 : 1)`，即**开机即置为“禁用”**，等 Grbl 进入 Idle 后由运动需要再使能。

## 9. 交付前检查清单

- **固件**：基于 `custom_3axis_hr4988.h`，含换纸系统 + M800 授权；烧录地址见前文（应用程序 `0x10000`）。仅烧固件不擦整片 Flash 时，NVS 保留，**无需重新授权**。
- **授权**：首次使用或整片擦除后需发 **M800 P121261 Q484**（或按 ChipID 向厂商索取对应 P/Q）。9 位数授权码必须用 P+Q 形式，单 P 会丢精度。
- **换纸**：第一页前可依赖 **G0 X0 Y0 Z0** 自动换纸，或先发 M721 / [ESP910]；页末由 **M30/M02** 自动换纸。若上位机不发 M30，需在每页末尾自行插入。
- **蓝牙**：固件默认 **蓝牙模式**，上电后 $I 会显示 `Mode=BT:Name=btgrblesp`（或当前蓝牙名）；需 WiFi 热点时在设置中将 Radio 改为 AP 并重启。
  - **蓝牙控制落笔抬笔**：**串口可以控制**说明固件与 Z 轴正常，固件对串口/蓝牙、XYZ 轴**都不区分**，同一套 `execute_line` 解析。  
  - **M3/M5 抬笔落笔**：本机型已开启 **M5=抬笔（Z20）、M3 S1000=落笔（Z0）**。蓝牙或串口发 `M5` 即抬笔，发 `M3` 或 `M3 S1000` 即落笔，无需发 G0 Z0/Z20。由 `custom_3axis_hr4988.h` 中 `USE_M3_M5_AS_PEN_UP_DOWN` 及 `PEN_UP_Z_MM`/`PEN_DOWN_Z_MM` 控制。  
  - **蓝牙下 XY 能控、Z 不能控**：若上位机发的是 M3/M5 而非 G0 Z，请用上述 M3/M5；若发 G0 Z 仍不能控，说明经蓝牙收到的行可能无 Z 字，可用串口监视器看 `[BT RX]` 回显排查。
  - **排查**：用蓝牙串口终端**手动发** `G0 Z0`、`G0 Z20`、`M5`（抬笔）、`M3 S1000`（落笔）或 `$J=Z5 F500`（记得带回车）；若能动则固件与蓝牙均正常。
  - **调试：蓝牙收包打印到串口**：固件已在 `Protocol.cpp` 中增加逻辑，当收到来自蓝牙的**完整一行**时，会将该行以 `[BT RX] 整行内容` 的形式发送到**串口（USB）**。用法：用 USB 连接设备并打开串口监视器，蓝牙端照常发指令；串口会打印每条蓝牙收到的行。若看到 `[BT RX] G0 Z0` 说明 Z 已送达固件；若只有 `[BT RX] G0 X10 Y20` 无 Z，则问题在上位机/蓝牙发送端。排查完成后可删除或注释掉该段代码（搜索 `[BT RX]`）。
- **已知限制**：换纸步进循环已加每 200 步 `delay(1)`，避免 ESP32 Interrupt Watchdog 复位；若出现 [PaperStatus] 3，表示第 7 步回找后传感器未稳定，多为纸路/传感器位置导致，可查 8.3。
- **文档**：本文档（配置.md）涵盖接线、命令、授权算法与故障排除；交付时一并提供即可。
