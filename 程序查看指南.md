# GRBL_ESP32 写字机程序查看指南

> 目标：帮助作者（和后来者）在较短时间内看懂整个工程，特别是自动换纸子系统的结构与流程。

---

## 一、整体工程结构从哪里看起？

- **工程入口**
  - `Grbl_Esp32/src/Grbl.cpp`：Grbl_ESP32 的主初始化和主循环。
  - `Grbl_Esp32/src/Config.h`、`Grbl_Esp32/src/Machine.h`：全局配置与机型选择。
- **写字机 + 换纸相关**
  - `Grbl_Esp32/src/Machines/custom_3axis_hr4988.h`：当前机型的 IO 映射与基础参数。
  - `Grbl_Esp32/src/IO配置参考文档.md`：整机 IO 对照表（强烈建议先通读一遍）。
  - `Grbl_Esp32/src/PaperChange*.{h,cpp}`：自动换纸完整实现。

**推荐总阅读顺序：**

1. `IO配置参考文档.md` —— 先把硬件引脚和电机/传感器搞清楚。
2. `Machines/custom_3axis_hr4988.h` —— 看当前机型如何把 ESP32 引脚映射到 GRBL 轴/限位等。
3. `Config.h` / `Machine.h` —— 理解哪些功能被编译进来了（如 `AUTO_PAPER_CHANGE_ENABLE`）。
4. `Grbl.cpp` —— 看主程序如何初始化、主循环是怎样调用换纸更新函数的。
5. `PaperChangeConfig.h` / `PaperChangeTypes.h` —— 看换纸子的“配置参数”和“数据结构”。
6. `PaperChange.h` / `PaperChange.cpp` —— 换纸系统的对外接口和总控。
7. `PaperChangeLogic.cpp` / `PaperChangeHardware.cpp` / `PaperChangeUtils.cpp` —— 深入每个状态和硬件驱动细节。

## 二、自动换纸子系统总览

### 2.1 相关文件一览

- **配置与类型**
  - `PaperChangeConfig.h`：换纸引脚、步数、时间、超时保护等参数统一定义。
  - `PaperChangeTypes.h`：状态机枚举、错误类型、全局控制结构等。
- **对外接口和总控**
  - `PaperChange.h`：对外暴露的 API（初始化、启动、一键换纸、更新、停止、调试函数等），已经补充详细中文注释。
  - `PaperChange.cpp`：持有全局状态结构，完成初始化、自检、入口函数等，充当“换纸子系统大脑”。
- **内部实现模块**
  - `PaperChangeLogic.cpp`：状态机 + 业务逻辑（PRE_CHECK / EJECTING / FEEDING / …）。
  - `PaperChangeHardware.cpp`：HC595、传感器、电机 GPIO 层面的具体驱动。
  - `PaperChangeUtils.cpp`：步进时序管理、电机通用控制、步数和毫米的换算等工具函数。

### 2.2 状态机大致流程

在 `PaperChangeTypes.h` 中有状态枚举，在 `PaperChangeLogic.cpp` 中有完整实现：

- `PAPER_IDLE`：空闲，等待 M 指令或按钮触发。
- `PAPER_PRE_CHECK`：预检面板上是否有纸，决定是需要先出纸还是直接进纸。
- `PAPER_EJECTING`：将当前纸张完全送出（A4 + 预留 4cm）。
- `PAPER_FEEDING`：送入新纸，直到传感器检测到纸张前沿。
- `PAPER_UNCLAMP_FEED`：松开压纸机构 + 使用送纸器送纸 4cm。
- `PAPER_CLAMP_FEED`：重新夹紧纸张。
- `PAPER_FULL_FEED`：进纸器 + 面板同步进给，直到纸张脱离传感器，然后面板单独再走 3cm。
- `PAPER_REPOSITION`：面板反转找边缘，再正转 3cm 完成精确定位。
- `PAPER_COMPLETE`：换纸完成，回到 `PAPER_IDLE`。
- `PAPER_ERROR`：错误/紧急停止状态。

`PaperChangeLogic.cpp` 顶部和各状态处理函数附近已经有较详细的中文注释，可以按状态顺序往下读。

## 三、如何顺着调用链看代码？

### 3.1 从主程序到换纸入口

1. 打开 `Grbl.cpp`，搜索 `AUTO_PAPER_CHANGE_ENABLE` 或 `paper_change_`：
   - 在初始化阶段会调用 `paper_change_init()`。
   - 在主循环中会周期性调用 `paper_change_update()`。
2. 再打开 `PaperChange.h`：
   - 看对外接口的中文注释，理解主程序“能对换纸系统做哪些操作”：
     - `paper_change_init()`：开机初始化。
     - `paper_change_start()` / `paper_change_one_click()`：启动自动换纸。
     - `paper_change_update()`：在主循环中驱动状态机。
     - `paper_change_stop()` / `paper_change_reset()`：错误和恢复。
     - `debug_*` 系列：配合 M80x 指令做硬件调试。
3. 进入 `PaperChange.cpp`：
   - 先看全局变量区（`paper_ctrl`、`last_error`、三路 `motor_timing`、静态标志）。
   - 然后按顺序看：
     - `paper_change_init()` → 硬件初始化、自检、结构体初始化。
     - `paper_change_start()` / `paper_change_one_click()` → 如何从 IDLE 切入状态机。
     - `paper_change_update()` → 如何读取传感器、驱动状态机、处理按钮。

### 3.2 状态机内部如何运作？

1. `PaperChangeTypes.h`：
   - 状态枚举 `paper_change_state_t`：每个状态旁都有简要中文注释。
   - 控制结构 `paper_change_ctrl_t`：看清状态机运行时需要维护哪些字段。
2. `PaperChangeLogic.cpp`：
   - 顶部：`is_valid_transition()` 和 `enter_state()` —— 了解状态转换是“查表 + 日志 + 重置标志”的封装逻辑。
   - `handle_state_machine_update()`：
     - 是真正的“调度中心”：根据当前 `ctrl->state` 调用对应的 `handle_xxx_state()`。
   - 依次阅读以下函数，就能把整套换纸流程在脑子里串起来：
     - `handle_pre_check_state()`
     - `handle_ejecting_state()` / `handle_full_ejection()`
     - `handle_feeding_state()`
     - `handle_unclamp_feed_state()`
     - `handle_clamp_feed_state()`
     - `handle_full_feed_state()`
     - `handle_reposition_state()`
     - `handle_complete_state()`
     - `handle_error_state()`
   - 每个函数前都有较长的中文注释，说明“物理意义 + 步数/距离依据 + 超时保护逻辑”。

### 3.3 硬件驱动和工具层

- `PaperChangeHardware.cpp`：
  - 从 `hc595_init()` 开始看，理解 Q0~Q7 分别连到了哪些电机/LED。
  - 注意批量更新器 `HC595BatchUpdater` 怎样减少写 HC595 的次数。
  - 向下继续看传感器读取、电机 GPIO 操作（配合 `IO配置参考文档.md` 理解）。
- `PaperChangeUtils.cpp`：
  - `init_motor_timing()` 和 `nonblocking_motor_step()` 是“全部电机非阻塞控制”的核心。
  - `steps_to_mm()` / `mm_to_steps()` 帮你在心中把“代码里的步数”转换成“真实毫米”。

## 四、理解配置参数与实际机械的对应关系

### 4.1 关键距离和步数

在 `PaperChangeConfig.h` 中：

- `PAPER_A4_LENGTH_MM`：A4 长度 297mm。
- `PAPER_4CM_LENGTH_MM`：进纸器额外送纸 4cm（文档中从 3.5cm 调整为 4cm）。
- `PAPER_3CM_LENGTH_MM`：3cm，用于传感器到夹紧机构距离 + 关键定位距离。
- `PAPER_1_5CM_LENGTH_MM`：1.5cm，用于压纸电机松开/夹紧的行程。
- 各自对应的 `*_STEPS` 常量会自动用 `DEFAULT_*_STEPS_PER_MM` 计算得到。

建议在看状态机代码时，随时对照这些宏，脑子里换算成“走了多少厘米/毫米”，会更直观。

### 4.2 超时与安全保护

- `PAPER_FEED_TIMEOUT_STEPS` / `PAPER_FEED_TIMEOUT_MS`：
  - 控制“进纸多久没检测到纸就认为异常”。
- `PAPER_MAX_REVERSE_STEPS`：
  - 限制 REPOSITION 反转找边缘的最大距离，防止过度反转损坏机构。
- `PAPER_STATE_TIMEOUT_MS`：
  - 单个状态最长允许时间，防止状态机卡死。

这些参数和逻辑保护，在 `PaperChangeLogic.cpp` 中通过宏 `CHECK_STATE_SAFETY(...)` 等方式体现。

## 五、如何从调试角度理解换纸流程？

### 5.1 M 指令与调试函数

在 `PaperChange.h` 里，已经用中文说明了几组常用调试接口，它们通常由 M80x 指令触发：

- `debug_feed_motor` / `debug_panel_motor` / `debug_clamp_motor`：
  - 用于逐个测试三路电机，确认方向、步数和行程是否正确。
- `debug_read_sensors`：
  - 一次性打印纸张传感器、一键换纸按钮状态等。
- `debug_emergency_stop` / `debug_reset_emergency`：
  - 人工触发/复位紧急停止。
- `debug_hc595_direct`：
  - 直接向 HC595 写入 8bit 数据，检查每一位实际控制到的硬件是否符合预期。

阅读顺序建议：

1. 先从 `debug_read_sensors()` 看传感器读法和日志格式。
2. 再看三个 `debug_*_motor()`，理解非阻塞步进和延时控制。
3. 合理搭配 `debug_hc595_direct()` 与 `IO配置参考文档.md` 做硬件连线验证。

### 5.2 日志体系

- `LOG_MSG` / `LOG_PROGRESS` / `LOG_ERROR` 等宏：
  - 在每个关键状态转移、超时、错误处都会打印一条人类可读的提示。
  - 阅读代码时可以把这些日志语句当做“时间轴上的里程碑”去理解整体流程。

## 六、如果只想快速知道“纸是怎么走完一圈”的？

1. 打开 `PaperChangeLogic.cpp`。
2. 只看以下几个函数的中文注释和关键代码：
   - `handle_pre_check_state()`
   - `handle_ejecting_state()`
   - `handle_feeding_state()`
   - `handle_unclamp_feed_state()`
   - `handle_clamp_feed_state()`
   - `handle_full_feed_state()`
   - `handle_reposition_state()`
   - `handle_complete_state()`
3. 配合 `PaperChangeConfig.h` 中的 `*_LENGTH_MM` 与 `*_STEPS`，在脑中把“步数”转成“纸的移动路径”。

读完这一圈，你基本就已经理解了 A4 纸在机器内部从“旧纸退出来”到“新纸定位好”的完整物理路径。

## 七、后续可以如何扩展或阅读？

- **想改机械参数**：优先改 `PaperChangeConfig.h` 中的长度/步数/间隔等宏，再观察状态机行为是否合理。
- **想加新的错误类型或恢复策略**：在 `PaperChangeTypes.h` 里扩展枚举 + 在 `PaperChangeLogic.cpp` 的错误处理处补逻辑。
- **想看 IO 更底层的时序**：继续深入 `PaperChangeHardware.cpp`，结合 `IO配置参考文档.md` 看每个 HC595 位对应的波形与引脚。

这份指南的目的，是让你以后回来看代码时，不需要再从零“扒逻辑”，而是按文件和状态机的结构直接“顺藤摸瓜”理解整个换纸系统和工程布局。
